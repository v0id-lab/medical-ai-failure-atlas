# Repo Acceleration North Star

Date: 2026 06 25

Status: public direction and build execution map.

## North star

Build a clinician led medical AI failure intelligence layer that turns model answers, benchmark claims, source claims, and public AI safety language into reviewable clinical risk signals without patient data or clinical use claims.

## Positioning

1. The repo is not another leaderboard.
2. The repo is not a clinical product.
3. The repo is not a cold outreach archive.
4. The repo is the public workbench for clinical severity, source support, claim hygiene, Turkish clinical context, and open source eval contribution discipline.

## Product stack

### 1. Clinician severity layer

Job: Turn a medical AI failure into a short clinician reason, a severity grade, and a safer review question.

Current Assets: Failure Atlas rows, clinician review protocol, safe failure cards, scoring rubric.

Next Build: Ship a generated severity layer spec with required fields and reviewer states.

Proof: A reviewer can judge one synthetic failure row without reading the full repo.

### 2. Source support layer

Job: Separate supported health claims from citation shaped text and missing evidence.

Current Assets: SourceCheckup rows, source review worksheets, source claim queue.

Next Build: Make source support fields visible in every public reviewer route.

Proof: A contributor can name the exact missing source support gap.

### 3. Claim hygiene layer

Job: Stop open comments, benchmark mentions, views, replies, and drafts from becoming endorsement or validation claims.

Current Assets: Public visibility claim gate, public GitHub route watchlist, public route preflight.

Next Build: Keep public preflight as the gate before any public route state is reused.

Proof: Every public route statement has a blocked claim check.

### 4. Turkish clinical context layer

Job: Translate global benchmark language into Turkish clinical wording risk, local workflow risk, and clinician literacy material.

Current Assets: TR MedLLM pack, Turkiye readiness kit, TEKNOFEST report checklist, Bilge readiness queue.

Next Build: Turn the scattered Turkish surfaces into a single path from scenario to review question to safe wording.

Proof: A Turkish medical AI team can use one page to avoid unsafe report language.

### 5. Open source eval bridge

Job: Convert the repo from a solo medical archive into a contributor path for lighteval, inspect ai, issue 154, and later lm eval routes.

Current Assets: lighteval pull request 1272, inspect ai pull request 4343, public route watchlist, review operating system.

Next Build: Publish the repo acceleration map and keep the PR open as the public handoff.

Proof: The README shows where a reviewer, maintainer, clinician, and open source eval contributor should enter.

## Strategic bets

### 1. Clinician judgment beats another benchmark

Why: Benchmark creation is crowded, but clinical severity judgment from a practicing physician is the defensible moat.

Six Month Result: A stable severity rubric and at least one public reviewer path that asks for risk, missing context, and safer wording.

Hours Version: Ship the severity layer product spec and connect it to the current review queue.

### 2. Small public objection beats broad outreach

Why: The repo needs external critique more than more internal artifacts.

Six Month Result: Issue 154 or successor route has repeatable outside objections and clear maintainer handling.

Hours Version: Expose the one sentence objection path and proof ledger in the acceleration docs.

### 3. Source support is the bridge between medicine and eval tooling

Why: Medical AI answers often sound useful while hiding missing source support, date risk, and guideline uncertainty.

Six Month Result: SourceCheckup becomes a visible reusable review lane tied to public contributor templates.

Hours Version: Make source support a required product stack layer and next queue lane.

### 4. Turkish clinical language is the local wedge

Why: Global health eval is crowded, but Turkish clinical wording, report hygiene, and local medical AI literacy are under served.

Six Month Result: A Turkish clinical AI safety mini path links scenario rows, reviewer questions, and report claim gates.

Hours Version: Add the Turkish context layer and three near term build tasks.

### 5. Open source proof matters more than ambition language

Why: The fastest credibility comes from small maintained public contributions and a repo that can be validated.

Six Month Result: At least two upstream routes, one maintained public issue route, and a README that shows the build system.

Hours Version: Push the acceleration system into PR 156 with validators and a README entry.
